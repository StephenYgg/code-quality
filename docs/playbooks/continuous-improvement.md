# Continuous Improvement Playbook

本手册用于在不要求历史代码一次性重写的前提下，建立质量基线、阻止新增退化并把真实事故转化为可执行规则。当前 Code Quality CLI 只实现 `CQ-AGENT-001` 的确定性验证切片；质量基线采集、评分比较、热点棘轮、事故复盘和规则生命周期仍先人工执行，直到对应命令、schema 和测试落地。

## 1. 建立质量基线

首次接入仓库时记录可复核的 `quality_baseline`：

- format、lint、typecheck、unit、integration、build、依赖和密钥检查的实际状态。
- 测试数量、执行时间、不稳定测试以及关键行为覆盖缺口。
- 大文件、大函数、宽 `try`、深嵌套、复杂条件、重复结果形状和高复杂度热点。
- `any`、ignore、suppression、deprecated 和 TODO 的数量与位置。
- 依赖漏洞、过期依赖、许可证、供应链和密钥风险。
- 缓存、任务、队列、timer、listener、stream、连接池和内存集合的容量、TTL、保留期与清理语义。
- 线上错误率、延迟、吞吐、任务积压、资源饱和和关键业务失败指标。
- 评分 scope、评分模型 ID/版本、profile 版本或内容哈希、权重、规则版本、舍入方式和证据位置。

基线不是豁免清单，也不能降低 P0/P1、密钥、数据完整性或危险并发的 Gate。它只用于区分历史债务和新增退化。`comparison_base` 只决定 Git diff；`benchmark_ground_truth` 只用于评测审查系统；三者必须分别命名和记录。

## 2. 评分与 Profile 比较纪律

质量趋势只比较相同 scope 的结果，例如 `change` 不能与 `repository` 混比。只有评分模型 ID 与版本相同、profile 版本或哈希及其权重兼容、适用项定义一致时，Score 和 Delta 才可直接比较。

以下任一变化都必须视为非等价比较，禁止把数值连成同一趋势：

- 新增、删除、重命名、重新锚定或重新加权评分项。
- 评分模型版本不同。
- profile 版本、内容哈希或 major/minor 权重不兼容。
- scope、适用性、coverage 口径或舍入规则不同。

非等价结果应标记为 `non-equivalent`，分别展示，必要时重新计算历史数据后再建立新基线。不得用 profile 或模型升级制造虚假的质量提升。`not_applicable` 必须有理由并从适用分母排除；`not_assessed` 是 coverage 缺口，不能伪装成满分。Gate 始终独立于 Score。

## 3. 增量棘轮

采用“保留基线、禁止恶化、持续收紧”：

- 新代码不得引入新的 P0/P1；配置要求阻断的 P2 也不得新增。
- 修改过的代码不得增加 lint、类型、安全、可读性或资源边界警告。
- 新增行为分支必须有对应的可信测试；关键并发交错应有可控测试。
- 修改已记录热点时，指标不得恶化，并应完成至少一项与本次变更直接相关的局部简化，前提是不扩大风险。
- 新增 ignore、suppression 或 waiver 必须有范围、原因、批准人、负责人、补偿控制和到期时间。
- 技术债不得无理由净增长；配置门槛只能在有证据、风险接受和明确批准时放宽。
- P0/P1、密钥泄露、数据损坏、危险并发放大和必选审查不完整不能被高分抵消。

棘轮比较应展示当前值、quality baseline、delta、证据和结论。历史低分不要求一次性重写，但不能成为继续恶化的理由。

## 4. 热点治理

按以下公式排序，而不是只按代码行数：

```text
治理优先级 = 变更频率 x 故障影响 x 理解难度 x 并发或安全风险
```

优先治理：

1. 经常修改且经常故障的核心路径。
2. 认证、跨租户、任务、缓存、事件和数据写入路径。
3. 单函数状态机、多副作用流程、宽错误边界和隐藏 fallback 优先级。
4. 缺少测试但承担公共 API、schema 或发布契约的模块。
5. 最后处理只影响风格、不影响行为或维护成本的历史问题。

治理前先补 characterization tests，再把状态解析、业务决策、副作用执行和结果呈现分离。对热路径建立 `峰值 QPS x 单请求扇出 x 保留时间` 的容量模型，并明确多实例协调、背压和资源上限。

## 5. 从事故和重复审查意见生成规则

每个重要事故、高严重度漏报或重复出现的审查意见都应完成以下闭环：

1. 记录真实触发条件、影响、时间线、根因和现有防线为何失效。
2. 判断能否转化为静态检查、行为测试、故障注入、benchmark 样本、报告字段或专项 checklist。
3. 提炼最小规则范围，给出稳定 ID、版本、风险说明、证据要求、通过条件、修复方向和正反例。
4. 把事故样本加入 `benchmark_ground_truth`，同时加入干净反例，衡量 precision、recall、误报、重复和稳定性；高严重度漏报必须单列。
5. 决定规则属于通用门禁、风险专项还是特定技术栈，不因单个案例扩大为噪声规则。
6. 更新测试、示例、文档和可观测性，使同类问题能在更早阶段被发现。

AI judging 可以辅助分类，但不能替代维护良好的人工标注评测集。评测均值不能掩盖 P0/P1 漏报。

## 6. 规则生命周期

每条新规则必须有真实案例、风险说明、scope、trigger、默认严重度、检测方式、证据要求、修复与验证方法、owner、示例和 lifecycle。

规则按以下阶段演进：

1. **Proposed**：说明要防止的失败和预期误报边界。
2. **Advisory**：以建议模式运行，收集命中、漏报、误报、重复、性能和维护成本。
3. **Blocking**：只有证据稳定、误报可接受、修复路径明确且执行成本可控时进入阻断。
4. **Deprecated**：存在替代规则、风险消失或技术栈退役时停止新增采用，并给出迁移方式。
5. **Removed**：记录删除原因、替代机制和生效版本，保留历史可追溯性。

规则语义变化必须升级版本并记录 changelog。至少每季度复查重复、过时、不可执行、与工具重叠或误报过高的规则。不得为了让当前变更通过而临时弱化全局规则。

## 7. 分阶段落地路线

### 阶段 0：盘点，不改行为

目标是掌握当前质量和风险分布。交付仓库结构与关键调用链、质量命令及实际结果、P0/P1 风险、高并发/认证/缓存/任务/数据路径清单、quality baseline 和前 10 个热点。完成条件是所有结论都有文件、命令、测试、日志或指标证据。

### 阶段 1：建立最低门禁

目标是阻止新增退化。逐步提供统一的 format、lint、typecheck、test、build、依赖和密钥检查，建立标准报告与 PR 模板，并让新增 lint、类型、构建、测试、密钥和 P0/P1 失败得到明确阻断。高风险变更必须填写专项审查。尚未实现的 CLI 或 CI 能力只能标为规划项。

### 阶段 2：治理核心热点

目标是降低最危险代码的修改成本。先补现有行为测试，再拆分大型流程，明确状态机、幂等键、锁、资源生命周期和单一 schema 来源，并对热路径建立 QPS、扇出和容量模型。完成条件是热点可独立理解、测试和验证，无需一次性全面重写。

### 阶段 3：扩大自动化

目标是把重复人工意见转成低误报工具。逐步加入增量覆盖率与复杂度、架构依赖与循环依赖、API/schema 兼容、并发/幂等/故障注入、性能基线和资源泄漏测试。自动化进入阻断前必须证明误报率、执行时间和维护责任可接受。

### 阶段 4：周期性复盘

目标是让规则随真实风险演进。每周整理重复意见和不稳定测试；每月更新热点、质量趋势和技术债净变化；每季度复查规则、评分模型、profile、误报、事故覆盖和门禁强度；每次事故后加入能提前发现同类问题的最小机制。

进入新阶段不代表上一阶段结束。每次变更仍需保持基线、棘轮、证据和独立 Gate，规划中的自动化也不能替代人工对系统行为与风险的判断。
