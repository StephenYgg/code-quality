# 严重级别、发现与豁免标准

本标准定义代码审查发现的严重级别、证据要求、生命周期、变更归属和豁免规则。严重级别衡量问题一旦成立所造成的影响；置信度衡量现有证据证明该问题成立的程度。两者必须独立记录，不得因影响严重而虚构高置信度，也不得因证据充分而提高影响级别。

评分只用于展示质量趋势和改进优先级。Gate 独立于评分运行，任何总分、维度分或历史增量都不能抵消达到阻断条件的发现。

## 严重级别

### P0：立即阻断

满足下列任一条件的已确认问题属于 P0：

- 可导致凭据、隐私或跨租户数据泄漏。
- 可导致不可恢复的数据损坏或大面积错误写入。
- 可被直接利用执行未授权操作或远程代码。
- 在正常生产流量下可能造成全局服务不可用。
- 已发生且仍在持续扩大的生产事故。

处理要求：禁止发布或合并，必须先修复并验证。P0 不得由评分、平均值或其他维度的良好表现抵消。

### P1：必须修复

典型情形包括：

- 主要用户路径行为错误。
- 并发竞争、重复副作用、锁失效或任务洪泛。
- 无界队列、无界缓存、无界重试或明显资源泄漏。
- 权限、幂等、事务或多实例语义错误。
- API 契约破坏且没有兼容或迁移方案。
- 测试无法覆盖或复现的高风险行为。

处理要求：当前变更合并前修复；只有用户明确接受风险、记录责任与控制措施，并形成有效豁免时才可例外。P1 不得由评分抵消。

### P2：应当修复

典型情形包括：

- 非主路径行为缺陷。
- 明显增加后续修改成本的职责混杂。
- 错误处理、类型契约或日志存在重要缺口。
- 性能会随数据量显著恶化，但尚未构成生产阻断。
- 关键行为有实现但没有可信测试。

处理要求：优先在当前变更中修复；若延期，必须记录负责人、原因和跟踪项。配置为阻断阈值的 P2 仍按 Gate 处理，不能由高分抵消。

### P3：改进建议

典型情形包括：

- 局部命名、结构、注释或测试表达可以更清楚。
- 不影响当前行为的轻度重复和维护成本。
- 可以逐步自动化的工程卫生问题。

处理要求：不阻断，但应评估是否可以低成本完成，并在重复出现时考虑沉淀为稳定规则。

### NIT：非阻断细节

- 仅限不影响理解、行为或维护成本的细节。
- 不得把个人偏好包装成 NIT 批量提交。
- formatter 或 linter 能稳定处理的问题不应占用人工审查主体。

处理要求：不阻断；不得将多个 NIT 人为汇总为更高严重级别。

## 发现数据模型

每条发现至少包含以下字段：

```text
id
rule_id
title
severity
confidence
status
disposition
locations
trigger
actual_behavior
expected_behavior
impact
evidence
remediation
verification
review_stage
provider
model
timestamps
```

字段约束如下：

- `id` 在一次审查及其后续复核中稳定且唯一。
- `rule_id` 指向可追溯的规则及版本；人工发现也必须使用明确的审查域或规则标识。
- `severity` 只能表达问题成立后的影响，不表达证据强弱。
- `confidence` 只能表达证据质量。置信度等级和阻断所需最低置信度由 profile 明确配置，不能从 `severity` 推断。
- `status` 表达验证生命周期；`disposition` 表达问题相对比较基线的归属。二者不得混用。
- `locations` 和 `evidence` 必须提供可复核锚点，不得只给结论。
- `provider`、`model` 和 `timestamps` 用于追溯产生、复核和更新结论的主体与时间。

## 生命周期

允许的主状态转换为：

```text
candidate -> corroborated -> confirmed -> reported
                        \-> dismissed
                        \-> uncertain
                        \-> waived
```

各状态的含义和处理规则如下：

- `candidate`：规则、静态信号或审查者提出了可复核疑点，但尚未完成上下文验证。不得作为已确认缺陷发布或阻断。
- `corroborated`：已有至少一项独立证据支持候选，例如调用链、测试、运行结果或第二审查阶段的同向结论，但仍需完成影响和触发条件确认。
- `confirmed`：触发条件、实际行为、预期行为、影响和证据链均已闭合，可以参与 Gate 判断。
- `reported`：已确认发现已进入面向作者的最终报告。报告必须保留其证据和归属。
- `dismissed`：证据证明候选不成立、不可触发或属于规则误报。必须记录驳回理由和反证，不能静默删除。
- `uncertain`：证据冲突、上下文缺失或环境不可复现，当前不能确认或驳回。必须列出缺失信息和下一步验证；不得伪装成 confirmed。
- `waived`：问题已确认，但有效豁免改变了 Gate 处置。发现仍须存储并在报告中可见，不得被抑制或改写为无问题。

阻断决定至少同时满足：状态为 `confirmed`，严重级别达到 profile 配置阈值，且所需置信度和适用范围满足配置。`candidate`、`corroborated` 或 `uncertain` 本身不构成已确认阻断，但高风险不确定项可使强制审查保持 `INCOMPLETE`。

## 变更归属

`disposition` 只能取以下值：

- `new`：由当前变更新增，或当前变更使既有问题恶化。
- `preexisting`：比较基线中已存在，且当前变更未使其恶化。
- `unknown`：缺少可靠比较基线或证据不足，无法判断归属。
- `not_applicable`：该规则在当前对象或范围内不适用，必须记录理由。

归属判断必须引用比较基线、变更行、符号级差异或指标增量。不得仅凭文件是否出现在 diff 中推断 `new`。一个问题可以是 `confirmed + preexisting`，也可以是 `uncertain + unknown`；状态和归属是正交维度。

## 证据与发现格式

每条对外发现应使用以下结构；不适用的字段也应明确说明，而不是省略关键推理：

```markdown
[P1] 简短、描述结果的标题

规则：`CQ-EXAMPLE-001@1`
状态：`confirmed`
置信度：`high`
归属：`new`
位置：`path/to/file.ts:123`，符号 `Example.run`

触发条件：在什么输入、状态、并发或部署条件下发生。

实际行为：代码现在会做什么。

预期行为：正确的系统行为是什么。

影响：用户、数据、安全、资源或兼容性影响。

证据：调用链、相关代码位置、测试结果、日志或复现命令及其关键输出。

修复方向：最小且可靠的修改方向，不强行指定唯一实现。

验证建议：需要新增或运行哪些测试。
```

证据必须尽量包含以下锚点：

- 位置锚点：仓库相对路径、行号和符号；多位置问题列出主位置与关联位置。
- 行为锚点：从输入到输出或副作用的调用链，注明关键分支和状态。
- 运行锚点：可复现命令、测试名称、关键输出和执行环境；未运行的验证明确标记“未运行”。
- 比较锚点：基线版本、diff、旧行为或指标 delta，用于判断 `new` 或 `preexisting`。
- 影响锚点：受影响用户、数据、资源、兼容性或安全边界，以及必要的规模估算。

禁止使用“这段代码不优雅”“建议优化一下”“可能有并发问题”“最好加点测试”“这里可以重构”等无法验证的表述。无法说明触发条件或影响时，应降低置信度，使用 `uncertain` 或保留为候选，并明确待确认信息。

## 豁免

每项豁免必须包含：

- 规则 ID 及适用版本范围。
- 仓库，以及路径、符号、变更或 finding 的精确作用域。
- 理由和风险接受声明。
- 批准人及负责整改或复核的 owner。
- 补偿控制。
- 跟踪项。
- 创建时间和到期时间。

缺少批准人、owner、到期时间或补偿控制的豁免无效；过期豁免自动失效。豁免不得扩大到未声明的仓库、路径、符号或后续版本。有效豁免只改变 Gate 处置，不删除发现、不降低严重级别、不提高置信度；报告和存储结果必须继续展示该发现及豁免状态。

## Gate 与评分

- Gate 依据已确认发现、适用的严重级别阈值、置信度要求、强制检查完整性和有效豁免独立计算。
- P0/P1、profile 配置为阻断的 P2、密钥泄露、数据损坏、不安全的并发放大、无效策略和未完成的强制审查保留各自 Gate 语义。
- 高分不能掩盖阻断发现；低分但无阻断发现只表示优先级和趋势风险。
- 报告必须并列展示 Gate、Score、baseline delta 和 confidence，不能只展示综合分。
