# 通用质量门禁

本标准适用于所有生产代码变更。审查者必须先理解变更意图和影响范围，再沿输入、状态、输出、副作用与失败路径追踪实际行为；不能只看 diff，也不能以格式、覆盖率或复杂度单项指标代替行为审查。

严重级别、证据要求和发现格式由 [严重级别与发现](severity-and-findings.md) 统一定义；完整审查步骤由 [审查流程](../playbooks/review-process.md) 统一定义。

## 意图与范围

必须检查：

- 变更是否解决了明确问题，变更前后的行为差异是否可说明、可验证。
- diff 是否混入无关重构、格式化、依赖升级或元数据噪声。
- 行为变化是否在说明、测试、schema 或公共接口中可见。
- 修改是否位于正确的层，是否绕过已有抽象或所有权边界。
- 是否存在更小、更直接且不削弱正确性的实现。
- 是否遗漏直接调用者、直接依赖、消费者、数据迁移、发布顺序或回滚路径。
- 对共享数据结构、缓存键、锁、持久化模型、外部 API、事件、任务和配置契约的影响是否已纳入范围。

以下情况默认阻断，直至风险被消除或按规则明确处置：

- 无法说明变更前后行为差异。
- 为局部需求引入一套平行架构。
- 在没有迁移和收敛方案时维护两个事实来源。

## 人类可读性

人类可读性、复杂度信号和热点增量棘轮只由 [人类可读性与可变更性](readability.md) 定义。本文件不复制其阈值或规则；所有生产代码审查仍必须执行该标准。

## 模块边界与架构

必须检查：

- 每个模块是否只有一个主要变化原因。
- 业务规则是否与协议、存储、日志和 UI 展示分离。
- 依赖方向是否符合项目架构，是否存在循环依赖、跨层访问或重复抽象。
- 共享状态是否有明确所有者、修改入口和生命周期。
- 公共接口是否小于内部实现表面，是否隐藏不稳定的数据库、SDK 和第三方类型。
- 外部 SDK、数据库和第三方 API 是否由稳定边界隔离。
- 新抽象是否减少真实复杂度或有意义的重复，而不是只增加文件和转发层。
- 兼容层是否有删除条件、期限和验证测试。

以下是阻断信号：

- 请求级凭据进入跨请求共享对象。
- 多个模块可以无约束修改同一状态。
- 一个模块同时决定业务状态、执行 I/O、持久化并构造多种展示结果。
- 为绕开旧架构创建第二套缓存、上下文、事件或错误体系。

## 行为正确性

必须检查：

- 正常路径、空输入、边界值和非法输入。
- `null`、`undefined`、空集合、零值和负值的业务语义。
- 时间、时区、精度、溢出、分页以及排序稳定性。
- 部分成功、超时、取消、重试和下游降级。
- 重复调用、乱序调用和迟到响应。
- 状态转换是否合法，终态是否可逆，恢复路径是否完整。
- 错误是否被错误吞掉、降级、重新分类或转换为成功结果。
- `try/catch` 是否捕获了本应阻断或由调用者处理的业务异常。
- 返回值、注释、日志、指标和测试表达的行为是否一致。
- 旧数据、旧客户端和旧配置是否仍按兼容承诺工作。

复杂状态机至少使用一种可审查表达：显式状态枚举与合法转换、转换表、分阶段函数，或覆盖全部状态与转换的参数化测试。仅把分支分散到多个函数不算显式状态模型。

## 状态、副作用与幂等

必须检查：

- 纯读取路径是否保持无副作用；查询不得暗中写状态、touch 数据或创建任务。
- 写操作的所有者、执行顺序和事务边界是否明确。
- 远程调用、事件、任务、邮件、扣费等副作用是否可安全重试且具备稳定、正确作用域和合理 TTL 的幂等键。
- 状态写入和事件发布是否可能一边成功一边失败；事务外副作用是否有 outbox、补偿或可恢复协议。
- 重试是否会重复创建记录、任务或外部资源。
- 删除、取消和补偿是否可以安全重复执行。
- 失败恢复是否依赖不可恢复的进程内状态。
- 生命周期结束时是否释放 timer、listener、lock、stream 和缓存条目。

凡涉及并发赢家、锁、任务、缓存或资源增长，还必须执行 [高并发与资源安全](concurrency.md)。

## 类型、接口与数据契约

必须检查：

- 公共边界不得使用无约束 `any`、无语义字典或开放索引签名。
- 输入在系统边界完成运行时校验，不在深层逻辑中猜测形状。
- 运行时 schema 与静态类型来自同一事实来源，或由一致性测试约束。
- 返回类型能够表达成功、失败、部分完成和可重试状态。
- 对外 DTO 不直接暴露内部数据库或 SDK 类型。
- 枚举、单位、时间格式和可选字段语义明确。
- schema 变化具备版本、消费者兼容和迁移策略。
- 类型断言不得绕过真实不确定性。
- `as unknown as`、非空断言和忽略类型错误必须有可验证理由和受限范围。

## 错误处理与可观测性

必须检查：

- 错误分类区分用户错误、业务错误、依赖错误和系统错误。
- 不把所有异常转换为同一个成功结果或通用错误。
- 对用户暴露的信息安全，对运维提供的信息足以诊断。
- 日志、指标或 trace 包含适用的 request、trace、session、job 或 entity 标识。
- 不记录 token、密钥、完整授权头、敏感正文或可重放凭据；详细要求见 [安全、隐私与 AI 审查运行安全](security.md)。
- 重试、降级、熔断、丢弃和最终失败有指标或结构化日志。
- 日志级别与实际严重度一致，同一异常不在多层重复记录制造噪声。
- 关键状态转换和最终结果可追踪。
- 告警基于用户影响或资源饱和，不只基于异常数量。

## 配置、兼容与发布

必须检查：

- 配置声明、读取、校验和运行时使用形成闭环。
- 默认值安全，且在不同环境中一致、可解释。
- 配置 getter 不暗中修改全局状态。
- 废弃配置有告警、迁移方式和删除计划。
- Feature flag 有所有者、默认值、生命周期和清理条件。
- API、事件和数据库变化保持向前与向后兼容，或明确版本化。
- 数据迁移可重入、可观察、可暂停，并有回滚或前滚方案。
- 发布顺序考虑多版本实例同时运行。
- 回滚旧版本不会读写其无法理解的新数据。
- 外部发布、通知、评论或远程写入均需符合 [安全标准中的外部副作用授权](security.md#外部副作用与发布)。

## 文档与仓库卫生

必须检查：

- README 只描述真实能力，不宣传尚未实现的功能。
- 架构文档与当前代码一致，并明确区分历史、当前和规划设计。
- 重复文档有自动同步机制或单一生成源。
- 构建、测试、lint 和本地运行命令可直接执行；尚未实现的命令必须显式标为规划，详见 [测试与自动化](testing-and-automation.md)。
- 生成目录、日志、临时文件和编辑器状态不进入版本控制。
- `.env`、密钥和私有证书不进入版本控制。
- 依赖、脚本和配置没有明显未使用项。
- 重要设计决策、限制和风险可被后续维护者找到。

### `CQ-AGENT-001`：同级 Agent 文档必须复用 `AGENTS.md`

这是本项目审查任何目标仓库时的正式验证规则，不只适用于本仓库。验证器必须同时枚举 `AGENTS.md` 与已知或配置的 peer Agent 文档，再按目录分组；不能只从已经存在 `AGENTS.md` 的目录出发，否则会漏掉只有 `CLAUDE.md`、`GEMINI.md` 或其他入口、却没有同级 canonical 文件的孤立作用域。内置 peer 名称包括 `CLAUDE.md`、`GEMINI.md`；profile 可以追加其他工具实际使用的文件名。

每个目录作用域必须满足：

- 共性 Agent 政策以同级 `AGENTS.md` 为唯一事实来源。
- 同级工具文档必须引用或指向 `AGENTS.md`，且只保留确有必要的工具特定差异。
- `symlink` 和仅包含强制读取指令的最小 Markdown 指针都是合法复用方式。
- 不得复制整段共性政策，避免副本随时间漂移。
- 工具特定差异不得削弱或否定同级 `AGENTS.md`。
- 引用关系不得形成循环；目标必须存在，并解析到预期目录作用域。

确定性检查必须报告：孤立 peer 作用域、缺失 `AGENTS.md` 引用、无法解析的指针、断开的 symlink、循环引用、规范化后完全或成段重复的共性内容，以及可以由结构化声明直接证明的冲突。语义近似、无法静态证明的重复或自然语言冲突只能产生人工或 AI 复核候选，不能伪装成确定性违规。

当前 `cq validate [repository]` 已实现本规则的首个 advisory 版本：内置识别 `CLAUDE.md` 与 `GEMINI.md`，检查孤立作用域、精确同级指向、read/follow 指令绑定、symlink 目标、Markdown 引用环、明确冲突、未归入工具增量的二级章节、规范化后的长段落精确复制，以及扫描、读取和 Markdown 解析资源上限。Markdown 在构建 AST 前限制为 10,000 行和 50,000 个语法标记，AST 遍历限制为 20,000 个节点；超限返回 `INCOMPLETE`。它不会对语义近似内容做确定性定罪，也尚未读取 profile 追加 peer 文件名或把告警提升为阻断；这些能力仍按设计渐进实现。

## 性能

性能审查以复杂度和资源模型为主，不以微优化为主。必须检查：

- 算法复杂度是否随真实数据量恶化。
- 循环内是否存在数据库或远程调用并形成 N+1。
- 是否重复解析、序列化、复制或读取大对象。
- 是否把可流式处理的数据全部加载到内存。
- 分页、批量和响应大小是否有合理硬上限。
- 连接池、超时、重试和并发上限是否协调一致。
- 缓存是否解决真实瓶颈，并具有失效、容量和一致性设计。
- 优化是否有 profile、benchmark 或生产指标证据。
- 优化是否牺牲正确性、隔离或可维护性。

无法说明数据规模和调用频率时，不得声称性能可以接受。涉及 QPS、扇出、缓存、队列或共享资源时，还必须执行 [高并发与资源安全](concurrency.md)。

## 风险触发矩阵

通用门禁始终执行；下表根据变更类型追加专项审查。一个变更命中多行时，所有对应检查叠加，不能由模型或审查者自行移除。

| 变更类型 | 必须追加检查 |
|---|---|
| HTTP/MCP/RPC 入口 | 认证、输入边界、兼容、限流、日志、热路径放大 |
| 数据库写入或迁移 | 事务、幂等、锁、索引、回滚、多版本部署 |
| 缓存 | 键空间、TTL、惊群、一致性、容量、多租户隔离 |
| 任务、队列、定时器 | 去重、并发上限、重试预算、背压、取消、回收 |
| 事件或消息 | at-least-once、乱序、重复、版本、死信、可观测性 |
| 认证与权限 | token 信任、资源归属、跨租户、日志脱敏、撤销 |
| 外部 API | 超时、重试、幂等、限流、降级、错误映射 |
| 文件和大数据 | 大小上限、流式处理、临时文件、清理、路径安全 |
| 公共 schema/API | 兼容、版本、消费者测试、弃用和迁移 |
| UI 状态与交互 | 空态、加载、错误、并发提交、可访问性、响应式 |
| 共享内存状态 | 生命周期、容量、清理、多实例语义、竞态 |

专项所有者如下：

- 热路径、任务、锁、缓存、事件、重试、共享状态和资源容量： [高并发与资源安全](concurrency.md)。
- 认证、权限、租户、输入、文件、URL、外部集成、密钥和 AI 运行边界： [安全、隐私与 AI 审查运行安全](security.md)。
- 测试、覆盖率、质量命令和自动化门禁： [测试与自动化](testing-and-automation.md)。
- 人类可读性与热点复杂度： [人类可读性与可变更性](readability.md)。
