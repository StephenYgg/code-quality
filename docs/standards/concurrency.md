# 高并发与资源安全

所有生产代码的设计、实现和修改都必须完成本标准的七项复核，而不只是在变更显式涉及“性能”时执行。某项确实不适用时，审查结论必须写明 `N/A` 及证据，不能静默省略。

低流量下行为正确、但在生产 QPS、多实例或失败重试下会产生任务洪泛、重复副作用、惊群、资源泄漏或全局不可用的设计，是代码缺陷，必须在发布前重构。优先采用原子 acquire-then-act；不得先创建副作用再依赖清理来争夺唯一赢家。

## 1. Hot path amplification

必须回答：

- 每个高 QPS 请求会创建多少 DB 行、缓存键、任务、事件、timer 和远程调用？
- 请求是否扇出为多个持久或异步资源？读取路径是否暗中 touch、写入或创建任务？
- 重试、分页、批量、回调或链式任务会不会继续放大扇出？
- 资源量是按请求增长，还是按有界的业务实体增长？

至少给出带单位的估算：

```text
峰值保留资源量 = 峰值 QPS x 单请求扇出 x 保留时间（秒）
峰值下游调用率 = 峰值 QPS x 单请求远程调用数
近似并发占用 = 到达率 x 平均占用时长
```

估算必须覆盖正常请求、最大合法批量和可重试失败路径。若保留资源随请求数而不是业务实体数增长，应改变数据模型、聚合方式或触发位置，不得只依赖事后清理。

## 2. Race / TOCTOU

必须检查：

- check-then-act 是否由原子操作、事务、唯一约束、compare-and-set 或等价机制保护。
- 多实例、进程重启和网络重试下是否仍只有一个合法赢家。
- 读改写是否会覆盖并发更新；版本号、条件更新或隔离级别是否足够。
- 超时、锁过期和迟到完成是否会破坏新状态。
- 旧 owner 是否可能更新新 owner 的结果或释放新 owner 的锁。
- 幂等记录、业务写入和外部副作用之间是否存在不可恢复的时间窗。

测试必须控制关键交错顺序并证明不变量；只并发启动多个 Promise 不能证明竞态安全。

## 3. Lock contention

必须检查：

- 锁键是否与业务唯一键一致，范围是否小于全仓库、全租户或全服务。
- 临界区是否只包含建立唯一性或提交状态所需的最短工作。
- 慢 I/O 是否必须在锁内；若移出锁外，是否会重新开放洪峰入口。
- 锁 TTL 是否覆盖最坏执行时间，长任务是否有安全续租。
- 等待者数量、等待时间、退避和取消是否有硬上限。
- 解锁是否校验不可猜测的 owner token，过期 owner 不得释放后继锁。
- 锁服务失效时是安全失败、降级还是重复执行，语义是否明确。

不能为缩短表面锁时长而在工作未完成时释放唯一入口；这会把锁从保护机制变成周期性洪峰开关。

## 4. Single-flight / dedupe

“每个 key 只执行一次”必须有可证明的单一赢家：

- winner 先原子占有执行权，再创建任务、事件、记录或远程副作用。
- loser 复用 winner 的结果、有限等待或明确退避，不得先执行再清理垃圾。
- winner 失败后允许按明确状态和预算安全重试。
- 去重键包含正确的租户、权限、版本和业务作用域，且有 TTL 与回收机制。
- 进程内 single-flight 只能协调单进程，不能冒充多实例保证。
- 外部发布、扣费、通知等还必须在目标系统或共享存储中具备幂等身份。

必须验证 loser 不留下孤儿记录、锁、任务、缓存键或重复外部结果。

## 5. Bounded background work

必须检查：

- 队列长度、worker 数、单批大小和并发数有硬上限。
- 重试具有次数或总时长预算、指数退避和抖动，不会永久循环。
- 下游变慢或不可用时有背压、拒绝、暂停或降级策略。
- 每请求 timer、listener、future 或后台链不会随流量无限增长。
- 链式任务的每一轮不会继续放大任务数。
- 取消、超时和关闭会终止子进程、请求及后续阶段。
- 失败和丢弃有可观察指标，队列饱和不会伪装成成功。

没有 daemon 或持久队列不代表工作天然有界；同步请求、Promise、子进程和重试同样必须受并发、数量、时间和输出预算约束。

## 6. Stampede / multi-instance

必须检查：

- 缓存同时过期时，所有实例是否会一起访问数据库或上游。
- 是否使用跨调用请求合并、随机过期、软过期、后台刷新或等价机制。
- 空结果与可安全复用的错误是否需要短期缓存，避免穿透。
- 缓存键是否包含租户、权限、版本和所有影响结果的参数。
- 本地缓存、本地锁和进程内状态是否被误认为全局协调。
- 多实例同时刷新、执行任务或发布结果时，共享唯一性和幂等性由谁保证。
- 缓存容量、TTL、淘汰、刷新失败和命中率是否可观察。

本地 single-flight 只能减少同一协调域内的重复工作。跨机器无法避免重复计算时，至少保证共享副作用幂等，并明确剩余的上游配额和成本风险。

## 7. Resource bounds

必须检查：

- Map、Set、数组、listener、stream、连接池、子进程和 timer 有容量上限与清理条件。
- 每租户、用户、session、task、key 和整个进程的最坏占用。
- 大响应、大输入、diff、批量、分页、日志和诊断输出有硬限制。
- 连接池、worker 池和限流器是否限制等待者堆积，而不只限制执行者。
- 保留期是否按业务实体设计，而不是按请求次数持续累积。
- 超限时是否返回显式拒绝、降级或 incomplete 状态，而不是静默截断为成功。
- 容量、饱和、驱逐、超时、拒绝和清理失败是否可监控。

资源结论必须给出峰值数量、保留时间、内存或存储上限以及超限行为。无法估算输入规模和调用频率时，不得声称资源安全。

## 本项目规划约束（尚未实现）

以下是已确认设计对未来完整 `code-quality` / `cq` 审查运行时的约束，不代表当前已实现 Provider、Hook、缓存、锁或模型审查；当前只有 README 所列的只读 `cq validate` 增量：

- 第一版不运行 daemon、服务端或脱离当前进程的后台队列。
- 完整审查的规划默认上限为 200 个变更文件、10,000 行变更、2 MiB 规范化 diff、七个审查阶段、两个并发 provider 请求、十六次总 provider 尝试、500,000 个总输入输出 token 和十五分钟墙钟时间。
- provider 工作满足 `峰值 provider 并发 <= 活跃审查数 x min(2, 可运行阶段数)`，单次审查总尝试不超过 16。
- 快速 Hook 预设必须使用更低的文件、token、请求和时间预算；超限产生明确的 reduced-scope 或 incomplete 结果。
- 运行和缓存写入采用原子创建与 rename；同一内容寻址审查键只有锁 winner 调用 provider，loser 有界等待或复用结果。
- 本地 lease 只协调共享同一状态目录的进程，不声明跨机器 single-flight。未来 CI 若启用分布式 worker，必须先增加共享锁或 check-run 唯一身份。
- 发布使用 `forge + repository + change number + head SHA + report hash` 作为幂等身份，并在发布前复核 head；跨机器可重复计算，但不得重复创建外部评论。
- 缓存必须配置容量和保留期并执行 LRU 清理；provider 限流和共享账号配额作为剩余风险报告。

## 标准输出

所有生产代码变更总结必须包含下列结构。七项均须填写；不适用项写 `N/A` 和理由。

```markdown
## Concurrency Review

- Hot path amplification: 请求到任务、事件、存储、timer 和远程调用的扇出，以及 QPS 估算。
- Race protection: 原子性、事务、唯一约束、CAS 或 owner-token 机制。
- Lock scope: 锁键、临界区、TTL、续租、等待上限和释放条件。
- Single-flight: winner、loser、去重键、失败重试和垃圾清理语义。
- Background bounds: 并发、批量、队列、重试、时间预算、取消和背压上限。
- Multi-instance behavior: 惊群保护、缓存失效和跨实例协调方式。
- Resource estimate: 峰值 QPS x 扇出 x 保留时间，以及内存、存储和连接上限。
- Residual risk: 尚未消除、需要监控或由外部系统约束的风险。
```
