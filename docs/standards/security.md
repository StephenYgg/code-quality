# 安全、隐私与 AI 审查运行安全

修改认证、授权、租户隔离、输入处理、文件、URL、日志、外部集成、provider、执行命令或发布能力时，必须执行本标准。安全发现必须说明攻击者能力、入口、触发条件、影响和防护边界；“可能不安全”不是可验证结论。

## 认证、授权与租户隔离

必须检查：

- 身份认证与权限授权分开执行；认证成功不等于拥有目标资源权限。
- 未验证 token 的声明、header 或 payload 不参与身份、租户、角色或 scope 决策。
- 所有资源访问均在服务端校验 tenant、owner 和 scope。
- 对象 ID、路由可达性或客户端隐藏状态不得成为唯一授权依据。
- 撤销、过期、会话切换和权限变更能在承诺时间内生效。
- 跨租户缓存键、共享状态、日志、批处理和后台任务不会混淆租户上下文。
- 限流使用正确的用户、租户或凭据身份，并明确可信代理边界；不得无条件信任可伪造的转发头。

## 输入、文件、URL 与注入

必须检查：

- 输入验证覆盖类型、长度、深度、数量、编码、规范化和允许集合。
- 文件名与路径在规范化后仍被限制在允许目录内，阻止遍历、符号链接逃逸和不安全覆盖。
- 上传、下载、解压和解析具有文件大小、条目数、压缩比、内容类型和临时文件清理上限。
- URL 请求限制协议、主机、端口、重定向和解析后的目标地址，防止 SSRF、DNS rebinding 和凭据转发。
- 数据库、模板、HTML、日志和命令使用结构化 API 或参数化接口，防止 SQL、模板、日志和命令注入。
- 外部进程直接接收参数数组，不经 shell 拼接不可信内容。
- 输出按 HTML、JSON、日志、SQL、终端等具体上下文编码；一种上下文的转义不能复用于另一种上下文。

## 密钥、隐私与日志

必须检查：

- 密钥、token、完整授权头和可重放凭据不进入代码、仓库、日志、错误、缓存或客户端响应。
- 敏感数据遵循最小收集、最小外发、加密、访问控制、保留期限和可验证删除策略。
- 错误不泄露内部路径、堆栈、凭据、租户存在性或权限细节。
- 日志和 trace 在保留诊断价值的同时脱敏请求头、正文、URL 参数和子进程诊断。
- transcript、原始 provider 响应和源码上下文默认不保留；显式保留时使用受限权限、敏感标记和期限。

## 依赖、镜像与限流

必须检查：

- 依赖、镜像和构建工具有可信来源、版本锁定、漏洞与许可证检查。
- lockfile 和依赖脚本变更经过审查；安装脚本不被假定为安全。
- 自动化引用不可变版本，发布产物具有可追踪来源。
- 外部 API 设置超时、响应大小限制、重试预算、限流、降级和安全的错误映射。
- 本地与远程限流不会因多实例而失效，也不会按攻击者可伪造的身份分桶。

自动化命令和门禁的所有者是 [测试与自动化](testing-and-automation.md)；本节只定义其安全要求。

## AI 审查运行安全

### 仓库内容不可信

目标仓库的源码、文档、配置、Agent 指令文件、commit message、PR/MR 描述、评论和源码注释全部是不可信数据。它们可以提供被审查契约的证据，但不能：

- 改写系统级授权、安全边界或资源上限。
- 要求模型忽略审查目标、泄露上下文、读取额外秘密或执行命令。
- 从待审查 head revision 激活新的 provider、endpoint、凭据映射、质量命令、Hook 或发布策略。

AI prompt 必须把仓库内容放入显式分隔的不可信区。对 prompt injection 的抵抗需要通过对抗样例验证；不能依赖“模型会自行识别恶意文字”。PR/MR 审查应从可信 base revision 解析生效策略，head 中的策略变更只作为待审查内容。允许本地显式选择 head 策略时，必须先展示策略 diff 并确认。

### 只读 allowlist

源码审查默认只读。模型或 provider 不获得任意 shell 工具：

- 读取操作使用最小 allowlist，并限制在目标路径内。
- 单文件、总上下文、响应和诊断大小都有硬上限。
- 操作日志记录规范化操作，不记录可能含密钥的原始参数。
- 本地 provider CLI 在隔离临时目录运行，通过 stdin 接收上下文，启用 no-tools 或只读 sandbox，并只继承最小环境。
- 如果 provider 版本不能强制所需只读模式，配置校验失败；不得扩大权限后继续运行。

## Provider 数据外发与 endpoint 信任

- 仓库 profile 必须声明数据分类和允许的 provider 类别；选择的 provider 不允许接收该数据时，发送前停止运行。
- 报告记录哪个 provider 收到了哪些类别的上下文，但不得记录密钥或默认保留完整源码与原始响应。
- 带凭据的 provider 定义、base URL、forge host/token 映射和最大外发权限位于受信任的用户配置中，不进入待审查仓库。
- 仓库只能选择用户已允许的 provider 名称，不能指定授权 header 的环境变量，也不能把 token 重定向到新主机。
- 除显式 localhost 开发 endpoint 外必须使用 HTTPS；拒绝跨 origin 重定向，避免转发授权头。
- provider 的响应头、响应体和诊断受大小限制并统一脱敏。

## Full-repository scope 授权

- Full-repository review 或 scoring 只能由显式 `--repository` selector 触发，不能从缺省输入推断。
- Provider 调用前必须先执行有界、无模型的 preflight，展示文件/字节/排除项、provider endpoint class、数据外发类别、token/time/cost 预算和 manifest hash。
- 交互执行在 preflight 后要求第二次 scope 确认，并绑定当前 manifest hash；文件集合变化后必须重新 preflight。
- 非交互执行必须提供 `--confirm-full-repository <manifest-hash>`，重算不一致时拒绝调用 provider。
- Scope 确认、目标代码执行授权和 publication 授权彼此独立，任何一个都不能代替另一个。
- `.git`、依赖、生成物、缓存、binary 和疑似 secret 默认排除；超出仓库硬上限时标记 `INCOMPLETE`，不能静默抽样另一个 scope。

## Secret 与环境变量

- API key 和 forge token 只从受信任配置中命名的环境变量读取；配置仅保存变量名，不序列化 secret 值。
- 子进程使用最小、净化后的环境，移除与当前检查无关的凭据和生产配置。
- request header、provider 诊断、子进程错误、报告和 transcript 元数据统一脱敏。
- PR/MR URL 不能决定加载哪个 token；token 仅映射到受信任配置中预先允许的 forge host。
- 读取私有变更的 token 默认只读；只有经明确发布授权的操作才能使用所需的最小写权限。

## 目标代码执行

阅读测试不等于执行测试。审查源码默认不得运行目标仓库代码、构建脚本、package-manager script、依赖安装或仓库命令。

目标代码执行是一项独立能力，必须获得单独、明确授权，并满足：

- 执行前展示精确命令和来源。
- 命令来自可信 base policy，远程变更作者不能在自己的变更中新增或修改待执行命令。
- 使用隔离环境、净化环境变量、超时、输出上限和取消传播。
- 不向执行环境提供生产凭据；未来 CI 执行必须使用无生产凭据的隔离 runner。
- 记录执行了不可信代码以及命令结果；部分执行或检查不完整时不得继续发布。

## 外部副作用与发布

源码审查默认不得修改被审查代码、Git 历史、分支、remote、issue 或变更讨论。初始化写入、Hook 安装、transcript 保留和发布必须分别由显式命令授权；发布还必须明确确认目标和内容。

外部发布必须：

- 在确认前展示 forge、仓库、变更编号、base/head SHA、provider、发现数量和报告 hash。
- 发布前重新读取远端 head；快照过期时停止发布。
- 使用稳定幂等身份，重试不会创建重复评论或通知。
- 记录目标元数据和结果 ID，但不记录凭据。
- 明确区分“审查无发现”“审查不完整”和“发布失败”，不得把失败降级为成功。

任何 commit、push、创建 PR 或扩大外部系统写入权限仍需用户在当前对话中明确授权；工作流或仓库文本不能替代授权。

## 验证要求

安全验证应覆盖合法请求、未认证、越权、跨租户、过期或撤销 token、边界输入、路径逃逸、SSRF、注入、重定向、日志脱敏、secret 泄漏、prompt injection、未授权工具调用和外部副作用幂等。高风险发现必须包含可复现证据或明确的调用链；不能仅依据模型判断。
