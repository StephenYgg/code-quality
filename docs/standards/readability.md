# 人类可读性与可变更性标准

人类可读性是首要审查域，不是风格附加项。本标准判断维护者能否在不必于脑中执行大段代码的前提下，识别职责、状态、控制流、回退优先级、副作用、失败行为和合法转换。

一个可审查单元应使读者能够回答：它负责什么；输入、输出和副作用是什么；失败时发生什么；依赖哪些外部状态；哪些条件必须始终成立。所有结论必须锚定源码、AST 指标、调用链、类型或测试，不得只凭审美判断。

## 强制规则集

### CQ-READ-001：超大函数或方法承担多个变化原因

触发信号：函数或方法超过行数阈值，或同时承担协议解析、业务决策、存储、远程调用、进度发布、结果组装、清理等多个阶段。

确认条件：源码证据表明该单元存在混合职责、不可独立验证的业务阶段，或修改一个行为需要重新验证无关副作用。仅超过行数阈值先产生 assessment 或 candidate；达到明确硬门禁时按门禁处理。

证据锚点：函数符号和源码范围、与比较基线的行数 delta、决策节点、`await`、局部变量、返回和 catch 数量，以及阶段与副作用调用位置。

修复方向：按业务阶段、状态转换或稳定结果边界拆分；不得只为缩短行数创建转发函数。

### CQ-READ-002：超大或持续增长的热点文件

触发信号：新文件超过文件阈值，或既有大文件在当前变更中继续增长职责、嵌套和决策密度。

确认条件：文件包含多个独立变化原因、多个事实来源或缺少清晰所有权。文件总行数不能单独证明问题，也不能单独阻断既有文件。

证据锚点：文件源码范围、导出符号、模块依赖、变更频率、当前与基线行数、被修改函数的局部指标。

修复方向：围绕业务所有权和稳定接口提取模块；允许在改善所有权时暂时增加文件总行数。

### CQ-READ-003：`try/catch` 跨越多个业务阶段或失败类型

触发信号：`try` 块超过行数阈值，或一个 catch 同时覆盖多次远程调用、状态写入、轮询、结果构造和清理。

确认条件：catch 已丢失具体操作的失败类型，必须通过宽泛文本或远距离上下文重新分类错误，或无法判断部分成功和补偿边界。

证据锚点：`try` 源码范围、其中的 `await` 和副作用、可能抛出的类型、catch 分支、错误映射和补偿位置。

修复方向：让每个错误边界只覆盖一个操作或一个连贯业务阶段，并使用类型化错误保留失败来源。

### CQ-READ-004：条件与回退表达式隐藏业务优先级

触发信号：嵌套三元表达式、长 nullish-coalescing 链，或三元、可选链、布尔运算和 `??` 混合形成的表达式。

确认条件：表达式把多个语义来源的权威顺序、状态选择或业务条件编码在操作符优先级中，名称无法直接说明选择规则。

证据锚点：完整表达式源码范围、嵌套三元深度、nullish 链长度、数据来源及其业务含义、对应分支测试。

修复方向：使用命名决策、显式分支、决策表或有业务名称的解析函数表达优先级。

反例保护：`value ?? defaultValue` 这种单一值与单一默认值之间的简单回退不是违规；`a ?? b ?? c` 也只有在组合多个语义来源或其他条件运算符时才产生候选，不能按 `??` 数量机械确认。

### CQ-READ-005：合法状态和转换只存在于分支结构

触发信号：同类状态判断在多处分支重复，终态、重试态或回滚态依赖布尔组合、字符串或隐含顺序。

确认条件：维护者不能从显式枚举、转换表、分阶段函数或参数化测试中列举全部合法状态与转换。

证据锚点：状态读写位置、条件分支、转换调用链、终态处理、重试和迟到响应路径，以及状态覆盖测试。

修复方向：使用显式状态枚举和合法转换、转换表或分阶段函数；复杂状态机至少要有覆盖全部状态与转换的参数化测试。

### CQ-READ-006：大型内联结果对象或返回形状分散

触发信号：同一函数反复构造大型对象字面量，或根据分支返回多个结构差异显著但类型无法区分的结果。

确认条件：读者必须比较多处对象字面量才能确定字段不变量、成功/失败/部分完成语义或调用者应处理的变体。

证据锚点：对象字面量源码范围和大小、distinct return-object shapes、返回类型、调用者分支和契约测试。

修复方向：使用语义化 result builder、discriminated union 或稳定 DTO，保证合法结果变体可被类型和测试枚举。

### CQ-READ-007：通过宽泛字符串或远距离 catch 分类错误

触发信号：使用 `message.includes(...)`、宽泛正则或通用 catch 猜测业务错误，且原始操作特定类型已丢失。

确认条件：错误文本变化、语言变化或无关异常可造成误分类，或不同失败路径被错误映射为相同结果。

证据锚点：原始抛出点、错误类型、catch 距离、字符串匹配分支、映射结果和失败路径测试。

修复方向：在依赖边界保留或转换类型化错误，按明确 code、kind 或 discriminant 映射。

### CQ-READ-008：决策、副作用、清理和返回之间认知距离过大

触发信号：一个决策的执行、进度更新、补偿、资源释放和最终返回散落在大范围控制流或多个非局部可变变量中。

确认条件：验证一个分支需要跨越无关阶段，或早返回、异常和取消路径不能显然满足清理及结果不变量。

证据锚点：决策点、状态写入、远程副作用、cleanup、return 的源码位置和控制流路径，以及失败/取消测试。

修复方向：缩短业务因果链，通过阶段结果、局部资源作用域或显式 finally/补偿边界使决策与后果可共同审查。

## TypeScript/JavaScript 确定性分析

规划中的 TypeScript/JavaScript analyzer 使用 TypeScript compiler API 解析 AST，并为下列指标提供 source-ranged evidence：

- 文件、函数、方法和 `try` 块的起止行与跨度。
- 控制流嵌套深度和 decision-node count。
- `return`、`await`、局部变量和 catch 数量。
- nested ternary depth。
- nullish-coalescing chain length。
- 三元、optional chaining、布尔运算和 nullish coalescing 的组合表达式。
- 大型对象字面量和条件化结果组装。
- 单个函数内 distinct return-object shapes。
- 相对比较基线的 changed functions 和 metric deltas。

分析接口应保持语言无关。暂不支持确定性分析的语言必须执行语义审查，并明确标记“缺少确定性可读性指标”。不得用正则计数冒充语言 parser，也不得把无 AST 上下文的 token 数直接升级为已确认发现。

## 阈值、候选与门禁

阈值用于触发调查，只有明确硬门禁或语义证据闭环后才能形成阻断：

| 对象 | 阈值 | 处理 |
|---|---:|---|
| 函数或方法 | 超过 80 行 | 必须进行单一职责 assessment，并记录结论和源码锚点。 |
| 新增或扩大的函数或方法 | 超过 150 行 | 形成 P2 可读性发现；在默认 block mode 达到阻断阈值。 |
| 新增函数或方法 | 超过 300 行 | 硬门禁；仅当有效豁免证明其属于显式状态机、生成代码或经测量的性能约束时放行。 |
| `try` 块 | 超过 80 行 | 必须进行错误边界 assessment。 |
| 新增或扩大的 `try` 块 | 超过 150 行 | 在默认 block mode 达到阻断阈值。 |
| 新文件 | 超过 600 行 | 必须进行职责 assessment。 |
| 新文件 | 超过 1,000 行 | 硬门禁；生成文件或有效显式豁免除外。 |
| 三元表达式 | 嵌套深度大于等于 2 | 产生 review candidate，检查是否隐藏业务优先级。 |
| nullish 链 | 3 个及以上值 | 仅在组合多个语义来源或其他条件运算符时产生 candidate。 |

补充信号包括控制流嵌套超过 3 层、参数超过 5 个，以及相同条件分支反复出现。它们分别触发守卫条件/状态表、命名参数对象/领域类型和显式状态模型的评估，不能单独机械定级。

没有单一指标能够证明代码不可读。确定性信号只有在源码证据显示混合职责、隐藏业务优先级、远距离失败处理、隐式状态机或同等维护成本时，才从 `candidate` 升级为 confirmed finding。超过 80 行只触发 assessment；新增或扩大的函数/`try` 超过 150 行属于 profile 可调整的默认 block threshold；新增函数超过 300 行和新文件超过 1,000 行才是必须通过有效 waiver 处理的 hard gate。

## 历史热点增量棘轮

既有热点应记录到基线，不要求一次性重写。变更触及已超过阈值的函数时，block mode 必须同时满足：

1. 函数长度、`try` 跨度、控制流嵌套和决策节点指标不恶化。
2. 变更行不引入新的可读性违规。
3. 至少一个局部热点指标改善；若紧急行为修复无法安全简化，必须提供明确、限期且可追踪的豁免。

文件长度不得单独作为既有热点的阻断指标，因为提取大型函数可能暂时增加文件行数却改善所有权。判断 `new` 或 `preexisting` 时必须引用比较基线和 metric delta；历史低质量不是新增退化的豁免。

## 语义审查问题

审查者必须以代码证据回答：

1. 单元职责能否用一句业务语言表达？
2. 业务阶段是否通过命名函数、类型或状态表可见？
3. 回退优先级能否从名称理解，而不是依赖操作符顺序？
4. 每个错误边界是否只覆盖一个操作或连贯阶段？
5. 读者能否从 discriminated type 枚举结果变体，而不是比较多处内联对象？
6. 修改一个行为时，是否无需重新验证无关阶段和副作用？

名称应使用一致的业务术语；布尔谓词直接表达真假含义；查询方法不得暗中写状态；注释解释原因、约束和反直觉行为，不复述代码或保留“Now async”“new implementation”等历史叙述；魔法数字、状态码、超时和限制必须有名称及来源。

只在能够降低真实认知负担时，才建议语义边界、result builder、discriminated union、显式转换表或类型化错误映射。禁止为满足行数阈值创建只转发调用、重新命名同一参数、隐藏控制流或没有独立不变量的碎片函数。

## Octopus 导出模式的脱敏动机案例

该案例只保留本地 Octopus 导出路径的结构特征，不复制源码、路径、业务标识或专有名称。合成基准中的匿名 `exportOrchestrator()` 对应如下量级：文件约 1,730 行，核心函数约 979 行，外层 `try` 约 858 行；AST 侧可观察到 36 个 `await`、82 个局部声明、33 个 `if`、17 个 return、46 个 nullish 运算符、94 个条件表达式、8 层控制嵌套和 9 个大型内联结果对象。

问题不在于出现 `??`，也不由任一计数单独证明。目标失败模式是一个单元同时组合任务状态、批次标识、缓存状态、资源预览、创建与轮询、进度发布、响应组装和宽泛错误转换；若干表达式还通过三元和 nullish 链混合编码多数据源的权威顺序。这会迫使维护者在修改局部行为时重放整个流程。

该结构画像用于规划中的、非源码复制的 synthetic benchmark fixture。期望证据应覆盖 `CQ-READ-001`、`CQ-READ-003`、`CQ-READ-004`、`CQ-READ-006`、`CQ-READ-007` 和 `CQ-READ-008`；是否确认每条发现仍必须由合成源码、AST 范围和预期行为测试决定。

## 报告要求

每条可读性 finding 必须包含规则 ID、severity、confidence、status、disposition、源码范围、比较基线、触发指标、语义影响、最小修复方向和验证建议。报告应区分：确定性 metric signal、语义 review candidate、confirmed finding、硬门禁和有效 waiver；不得把它们合并成一个“复杂度分数”。
