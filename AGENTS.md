# Code Review Quality System

本仓库用于建立、验证和持续改进一套可执行的生产代码审查体系。

这里的目标不是收集个人风格偏好，也不是生成一份永远不变的检查清单。
目标是让代码审查能够持续发现真实风险、提供可验证证据、推动增量改进，
并把重要经验沉淀为自动化门禁或明确规则。

本文件适用于在本仓库中工作的所有开发者和 AI Agent。

---

## 1. 指令优先级与工作边界

按以下顺序执行指令：

1. 当前对话中用户的明确要求。
2. 目标仓库中距离被审查文件最近的 `AGENTS.md` 或等效项目规则。
3. 本文件。
4. 工具、框架和语言的通用最佳实践。

如果目标仓库规则与本文件冲突，必须指出冲突，并优先遵守更具体、更高优先级的规则。

### 1.1 Git 提交权限

- 未经用户在当前对话中明确授权，禁止执行 `git commit`、创建提交、推送或创建 PR。
- “开始”“继续”“修复”“完成”“测试通过”都不等于提交授权。
- 可以给出建议提交信息，但不得代替用户提交。
- 工作区可能包含用户未提交的改动。不得回滚、覆盖或格式化无关改动。

### 1.2 审查与修改的边界

- 用户要求“审查、分析、解释”时，默认只读，不修改生产代码。
- 用户要求“修复、实现、重构”时，才可以进行相应修改。
- 审查中发现的问题不得被静默修复。先报告，再依据用户授权处理。
- 不为展示审查能力而扩大范围。只追踪与当前变更、调用链和风险面相关的代码。

### 1.3 证据原则

- 结论必须基于实际代码、配置、测试或运行结果。
- 不得用“看起来没问题”“应该能工作”代替验证。
- 每个问题尽量给出文件、行号、触发条件、影响和修复方向。
- 自动化检查结果只是证据的一部分，不能代替行为和架构审查。
- 没有运行的测试必须明确标为“未运行”。

## 2. 项目使命

本项目的长期使命包括：

1. 定义统一、可解释、可执行的代码审查标准。
2. 把严重问题与普通建议分开，减少无效争论。
3. 把生产事故、线上退化和高频审查意见沉淀为规则。
4. 将稳定、机械、低误报的规则逐步自动化。
5. 用增量治理改善历史代码，而不是要求一次性重写。
6. 保证新代码不扩大已知技术债，并持续收紧质量基线。
7. 让审查结论对作者、审查者和后续维护者都可复核。

### 2.1 不追求的目标

- 不追求所有文件使用相同写法。
- 不把审查变成格式、命名和个人偏好的争论。
- 不以代码行数、圈复杂度或覆盖率单个指标决定质量。
- 不要求为了“架构漂亮”进行与需求无关的大规模重构。
- 不通过增加抽象层掩盖复杂度。
- 不允许用平均分掩盖安全、数据损坏或并发阻断问题。

## 3. 核心审查模型

所有审查采用两层模型：

1. **通用质量门禁**：适用于所有生产代码。
2. **风险专项检查**：按变更涉及的热路径、任务、缓存、认证、数据迁移等风险触发。

审查步骤必须是：

```text
理解意图 -> 确认范围 -> 标记风险 -> 追踪行为 -> 检查失败路径
-> 检查并发与资源 -> 检查测试 -> 运行验证 -> 输出结论
```

### 3.1 审查对象

审查不能只看 diff。至少要检查：

- 被修改代码。
- 直接调用者和直接依赖。
- 共享数据结构、缓存键、锁和持久化模型。
- 对外 API、事件、任务和配置契约。
- 对应测试和运维可观测性。
- 变更前后的兼容性与迁移路径。

### 3.2 评分与门禁

评分只用于趋势观察，不用于决定能否合并。任何 P0 或 P1 问题都不能被其他维度的高分抵消。

可选评分维度，每项 0 到 5 分：

- 可读性与局部清晰度。
- 模块边界与架构一致性。
- 行为正确性。
- 并发与资源安全。
- 安全与隐私。
- 测试可信度。
- 运维与可观测性。
- 文档和变更完整性。

评分必须附证据。不得只给数字。

## 4. 严重级别

### P0: 立即阻断

满足任一条件即为 P0：

- 可导致凭据、隐私或跨租户数据泄漏。
- 可导致不可恢复的数据损坏或大面积错误写入。
- 可被直接利用执行未授权操作或远程代码。
- 在正常生产流量下可能造成全局服务不可用。
- 已发生且仍在持续扩大的生产事故。

处理要求：禁止发布或合并，必须先修复并验证。

### P1: 必须修复

典型情形：

- 主要用户路径行为错误。
- 并发竞争、重复副作用、锁失效或任务洪泛。
- 无界队列、无界缓存、无界重试或明显资源泄漏。
- 权限、幂等、事务或多实例语义错误。
- API 契约破坏且没有兼容或迁移方案。
- 测试无法覆盖或复现的高风险行为。

处理要求：当前变更合并前修复，除非用户明确接受并记录风险。

### P2: 应当修复

典型情形：

- 非主路径行为缺陷。
- 明显增加后续修改成本的职责混杂。
- 错误处理、类型契约或日志存在重要缺口。
- 性能会随数据量显著恶化，但尚未构成生产阻断。
- 关键行为有实现但没有可信测试。

处理要求：优先在当前变更中修复；若延期，必须有负责人、原因和跟踪项。

### P3: 改进建议

典型情形：

- 局部命名、结构、注释或测试表达可以更清楚。
- 不影响当前行为的轻度重复和维护成本。
- 可以逐步自动化的工程卫生问题。

处理要求：不阻断，但应评估是否可以低成本完成。

### NIT: 非阻断细节

- 仅限不影响理解、行为或维护成本的细节。
- 不得把个人偏好包装成 NIT 批量提交。
- 可由 formatter 或 linter 处理的问题，不应占用人工审查主体。

## 5. 标准问题格式

每条发现必须尽量包含：

```markdown
[P1] 简短、描述结果的标题

位置：`path/to/file.ts:123`

触发条件：在什么输入、状态、并发或部署条件下发生。

实际行为：代码现在会做什么。

预期行为：正确的系统行为是什么。

影响：用户、数据、安全、资源或兼容性影响。

证据：调用链、代码片段、测试结果、日志或复现命令。

修复方向：最小且可靠的修改方向，不强行指定唯一实现。

验证建议：需要新增或运行哪些测试。
```

禁止使用以下模糊意见：

- “这段代码不优雅。”
- “建议优化一下。”
- “可能有并发问题。”
- “最好加点测试。”
- “这里可以重构。”

如果无法说明触发条件或影响，应降低确定性并标明待确认信息。

## 6. 通用质量门禁

### 6.1 意图与范围

检查项：

- 变更是否解决了明确问题。
- diff 是否包含与目标无关的重构、格式化或依赖升级。
- 行为变化是否在说明、测试或接口中可见。
- 是否修改了错误的层，或绕过已有抽象。
- 是否有更小、更直接的实现方式。
- 是否遗漏调用者、消费者、迁移或回滚路径。

阻断信号：

- 无法说明变更前后行为差异。
- 为一个局部需求引入新的平行架构。
- 在没有迁移方案时同时维护两个事实来源。

### 6.2 人类可读性

代码应允许读者回答：

1. 这段代码负责什么？
2. 输入、输出和副作用是什么？
3. 失败时会发生什么？
4. 它依赖哪些外部状态？
5. 哪些条件必须始终成立？

检查项：

- 名称使用业务语言，而不是模糊的 `data`、`info`、`handle`、`process`。
- 布尔变量和谓词能直接表达真假含义。
- 查询方法不应暗中写状态；有副作用的方法应从名称体现。
- 控制流按业务阶段排列，优先使用早返回减少嵌套。
- 同一概念在代码、配置、日志和文档中使用同一术语。
- 注释解释原因、约束和反直觉行为，不复述代码。
- 删除 `Now async`、`new implementation` 等只描述历史的注释。
- 魔法数字、状态码、超时和限制有命名及来源说明。
- 错误和返回类型能表达所有合法状态。
- 读者无需跨越大量文件才能确认一个简单行为。

复杂度信号，不是机械裁决：

- 函数超过 80 行：要求说明为何仍是单一职责。
- 函数超过 150 行：默认应拆分，保留必须有明确状态机或性能理由。
- 文件超过 600 行：检查是否混入多个变化原因。
- 文件超过 1000 行：必须提供拆分或保持现状的书面判断。
- 嵌套超过 3 层：优先使用守卫条件、辅助函数或状态表。
- 参数超过 5 个：评估使用命名参数对象或领域类型。
- 同类条件分支反复出现：评估状态模型是否缺失。

不得为了满足行数阈值而创建只转发调用、没有语义价值的碎片函数。

### 6.3 模块边界与架构

检查项：

- 每个模块是否只有一个主要变化原因。
- 业务规则是否与协议、存储、日志、UI 展示分离。
- 依赖方向是否符合项目架构。
- 共享状态是否有明确所有者和生命周期。
- 公共接口是否小于内部实现表面。
- 是否通过稳定接口隔离外部 SDK、数据库和第三方 API。
- 是否存在循环依赖、跨层访问或重复抽象。
- 新抽象是否减少真实复杂度，而不是只增加文件数。
- 兼容层是否有删除条件、期限和测试。

阻断信号：

- 请求级凭据进入跨请求共享对象。
- 多个模块可以无约束修改同一状态。
- 一个模块同时决定业务状态、执行 I/O、持久化并构造多种展示结果。
- 为绕开旧架构创建第二套缓存、上下文、事件或错误体系。

### 6.4 行为正确性

检查项：

- 正常路径、空输入、边界值和非法输入。
- `null`、`undefined`、空集合、零值和负值语义。
- 时间、时区、精度、溢出、分页和排序稳定性。
- 部分成功、超时、取消、重试和下游降级。
- 重复调用、乱序调用和迟到响应。
- 状态转换是否合法，终态是否可逆。
- 错误是否被错误地吞掉、降级或重新分类。
- `try/catch` 是否意外捕获本应阻断的业务异常。
- 返回值、注释、日志和测试表达的行为是否一致。
- 旧数据、旧客户端和旧配置是否仍能按承诺工作。

任何复杂状态机都应具备以下至少一种表达：

- 显式状态枚举和合法转换。
- 转换表。
- 分阶段函数。
- 覆盖所有状态与转换的参数化测试。

### 6.5 状态、副作用与幂等

检查项：

- 纯读取路径是否保持无副作用。
- 写操作的所有者、顺序和事务边界是否明确。
- 远程调用、事件、任务、邮件、扣费等副作用是否幂等。
- 幂等键是否稳定、作用域正确并有合理 TTL。
- 状态写入和事件发布是否可能一边成功一边失败。
- 重试是否会重复创建记录、任务或外部资源。
- 删除、取消和补偿是否可以安全重复执行。
- 失败恢复是否依赖不可恢复的进程内状态。
- 生命周期结束时是否释放 timer、listener、lock、stream 和缓存条目。

### 6.6 类型、接口与数据契约

检查项：

- 公共边界不使用无约束 `any`、字典或开放索引签名。
- 输入在系统边界验证，而不是在深层逻辑中猜测。
- 运行时 schema 与静态类型来自同一事实来源，或有一致性测试。
- 返回类型表达成功、失败、部分完成和重试状态。
- 对外 DTO 不直接暴露内部数据库或 SDK 类型。
- 枚举、单位、时间格式和可选字段语义明确。
- schema 变化有版本、兼容和迁移策略。
- 类型断言不会绕过真实不确定性。
- `as unknown as`、非空断言和忽略类型错误必须说明原因。

### 6.7 错误处理与可观测性

检查项：

- 错误分类区分用户错误、业务错误、依赖错误和系统错误。
- 不把所有异常转成同一个成功或通用错误结果。
- 错误信息对用户安全，对运维足够诊断。
- 日志包含 request、trace、session、job 或 entity 标识。
- 不记录 token、密钥、完整授权头、敏感正文和可重放凭据。
- 重试、降级、熔断和丢弃行为有指标或日志。
- 日志级别与实际严重度一致。
- 不在多个层重复记录同一异常并制造噪声。
- 关键状态转换和最终结果可追踪。
- 告警基于用户影响或资源饱和，不只基于异常数量。

### 6.8 测试质量

检查项：

- 测试验证对外行为，而不是复制实现步骤。
- 测试名称描述条件和预期结果。
- 修复缺陷时先有能复现原问题的测试。
- 覆盖成功、失败、边界、重试、取消和并发路径。
- 时间、随机、网络和外部状态可注入或可控制。
- 测试之间不依赖执行顺序或共享脏状态。
- Mock 保留真实契约，不因过度简化而掩盖问题。
- 集成测试覆盖关键边界，单元测试覆盖分支和不变量。
- 并发测试能控制交错顺序，而不是只启动多个 Promise。
- 测试失败能快速定位行为，而不是依赖数千行共享 fixture。

覆盖率规则：

- 覆盖率是发现遗漏的信号，不是目标本身。
- 新增关键分支必须有行为测试。
- 不允许用无断言、只执行代码的测试提高覆盖率。
- 历史低覆盖模块采用增量棘轮，修改过的代码不得降低覆盖。

### 6.9 配置、兼容与发布

检查项：

- 配置声明、读取、校验和运行时使用形成闭环。
- 默认值安全，并在不同环境中一致可解释。
- 配置 getter 不暗中修改全局状态。
- 废弃配置有告警、迁移方式和删除计划。
- Feature flag 有所有者、默认值、生命周期和清理条件。
- API、事件和数据库变化保持向前与向后兼容，或明确版本化。
- 数据迁移可重入、可观察、可暂停并有回滚或前滚方案。
- 发布顺序考虑多版本实例同时运行。
- 回滚旧版本不会读写无法理解的新数据。

### 6.10 文档与仓库卫生

检查项：

- README 描述真实能力，不宣传尚未存在的功能。
- 架构文档与当前代码一致，并注明历史设计。
- 重复文档有自动同步或单一生成源。
- 构建、测试、lint 和本地运行命令可直接执行。
- 生成目录、日志、临时文件和编辑器状态不进入版本控制。
- `.env`、密钥和私有证书不进入版本控制。
- 依赖、脚本和配置没有明显未使用项。
- 重要设计决策、限制和风险可被后续维护者找到。

## 7. 高并发与分布式专项审查

任何热路径、任务、缓存、锁、事件、重试、后台工作或共享状态变更都必须执行本节。
低流量正确但生产流量危险，属于代码缺陷。

### 7.1 热路径放大

必须回答：

- 每个请求会创建多少 DB 行、缓存键、任务、事件和远程调用？
- 是否存在请求到资源的扇出？
- 是否把纯查询变成隐式写入、touch 或任务创建？
- QPS 上升时，资源量是否按请求增长而不是按业务实体增长？

至少给出估算：

```text
峰值资源增长 = 峰值 QPS x 单请求扇出 x 保留时间
```

### 7.2 Race 与 TOCTOU

检查项：

- check-then-act 是否由原子操作、事务、唯一约束或 compare-and-set 保护。
- 多实例是否可能同时成为赢家。
- 读改写是否会覆盖其他并发更新。
- 超时、锁过期和迟到完成是否会破坏新状态。
- 旧 owner 是否可能释放新 owner 的锁。

优先使用 acquire-then-act，而不是 create-then-cleanup。

### 7.3 锁与临界区

检查项：

- 锁粒度是否与业务唯一键一致。
- 临界区是否包含不必要的慢 I/O。
- 锁 TTL 是否覆盖最坏执行时间，并考虑续租。
- 等待者是否有上限、超时和退避。
- 锁释放是否校验 owner token。
- 工作未完成前释放锁是否会重新打开洪峰入口。

### 7.4 Single-flight 与去重

检查项：

- “每个 key 只执行一次”是否有真实的单一赢家。
- 失败后是否允许安全重试。
- loser 是否复用结果或有明确退避，而不是重复执行。
- 进程内 single-flight 是否被误认为多实例协调。
- 去重记录是否有作用域、TTL 和回收机制。

### 7.5 后台工作有界性

检查项：

- 队列、worker、批量大小和并发数有硬上限。
- 重试有次数或总时间预算，并包含抖动。
- 下游变慢时有背压、拒绝或降级。
- 定时器数量不随请求无限增长。
- 链式任务不会每轮继续放大。
- 取消和关闭会停止后续工作。

### 7.6 缓存与惊群

检查项：

- 缓存过期是否会让所有实例同时访问 DB 或上游。
- 是否使用请求合并、随机过期、软过期或后台刷新。
- 空结果和错误是否需要短期缓存。
- 本地缓存与全局一致性要求是否匹配。
- 缓存键包含租户、权限、版本和影响结果的参数。
- 缓存容量、TTL、淘汰和监控明确。

### 7.7 资源边界

必须检查：

- Map、Set、数组、listener、stream、连接池和 timer 是否有上限与清理。
- 每个租户、用户、session、task 和 key 的最坏占用。
- 大响应、大输入、批量和分页是否有硬限制。
- 连接池和 worker 池是否会造成等待者堆积。
- 保留期是否基于业务实体，而不是请求次数。

### 7.8 并发审查输出

涉及并发时，变更总结必须包含：

```markdown
## Concurrency Review

- Hot path amplification: 请求到任务、事件、存储和远程调用的扇出。
- Race protection: 原子性、事务、唯一约束或 owner-token 机制。
- Lock scope: 锁键、临界区、TTL、续租和释放条件。
- Single-flight: winner、loser 和失败重试语义。
- Background bounds: 并发、批量、队列、重试和背压上限。
- Multi-instance behavior: 跨实例协调方式。
- Resource estimate: 峰值数量、保留时间和内存或存储上限。
- Residual risk: 尚未消除且需要监控的风险。
```

## 8. 安全与隐私专项审查

修改认证、授权、租户隔离、输入处理、文件、URL、日志或外部集成时必须检查：

- 身份认证与权限授权是否分开。
- 未验证 token 的内容不得参与安全决策。
- 所有资源访问都在服务端验证 tenant、owner 和 scope。
- 对象 ID 不得成为唯一授权依据。
- 输入验证覆盖长度、类型、深度、编码和允许集合。
- 文件路径防止遍历，URL 请求防止 SSRF。
- 数据库查询、模板和命令执行防止注入。
- 输出按 HTML、JSON、日志、SQL 等具体上下文编码。
- 密钥不进入代码、日志、错误、缓存和客户端响应。
- 敏感数据有最小化、加密、保留和删除策略。
- 错误信息不泄露内部路径、堆栈、凭据和权限细节。
- 限流基于正确身份和代理信任模型。
- 依赖和镜像有来源、锁定与漏洞检查。

安全问题必须描述攻击者能力、入口、影响和防护边界，不接受只写“可能不安全”。

---

## 9. 性能专项审查

性能审查以复杂度和资源模型为主，不以微优化为主。

检查项：

- 算法复杂度是否随真实数据量恶化。
- 循环内是否存在数据库或远程调用，形成 N+1。
- 是否重复解析、序列化、复制或读取大对象。
- 是否把可流式处理的数据全部加载到内存。
- 分页、批量和响应大小有合理上限。
- 连接池、超时、重试和并发上限协调一致。
- 缓存是否解决真实瓶颈，并包含失效与一致性设计。
- 优化是否有 profile、benchmark 或生产指标证据。
- 性能优化是否牺牲正确性、隔离或可维护性。

无法说明数据规模和调用频率时，不得声称性能可以接受。

## 10. 风险触发矩阵

根据变更内容追加专项审查：

| 变更类型 | 必须追加检查 |
|---|---|
| HTTP/MCP/RPC 入口 | 认证、输入边界、兼容、限流、日志、热路径放大 |
| 数据库写入或迁移 | 事务、幂等、锁、索引、回滚、多版本部署 |
| 缓存 | 键空间、TTL、惊群、一致性、容量、多租户隔离 |
| 任务、队列、定时器 | 去重、并发上限、重试预算、背压、取消、回收 |
| 事件或消息 | at-least-once、乱序、重复、版本、死信、可观测性 |
| 认证与权限 | token 信任、资源归属、跨租户、日志脱敏、撤销 |
| 外部 API | 超时、重试、幂等、限流、降级、错误映射 |
| 文件和大数据 | 大小上限、流式处理、临时文件、清理、路径安全 |
| 公共 schema/API | 兼容、版本、消费者测试、弃用和迁移 |
| UI 状态与交互 | 空态、加载、错误、并发提交、可访问性、响应式 |
| 共享内存状态 | 生命周期、容量、清理、多实例语义、竞态 |

## 11. 自动化质量门禁

优先将确定性强、误报低、可快速执行的规则自动化。

### 11.1 推荐的基础命令

每个目标仓库应逐步提供统一命令，名称可按技术栈调整：

```text
format:check
lint
typecheck
test:unit
test:integration
test:concurrency
build
check:dependencies
check:secrets
```

### 11.2 合并前最低门禁

- 格式检查通过。
- lint 无新增错误。
- 类型检查通过。
- 受影响单元测试通过。
- 关键集成测试通过。
- 构建通过。
- 无密钥和敏感文件进入 diff。
- P0/P1 为零。
- P2 已修复或有明确的延期记录。
- 高风险变更包含专项审查结论。

### 11.3 自动化边界

- 不用静态指标自动阻断所有大函数或大文件。
- 不用覆盖率数字替代测试质量判断。
- 不允许通过全局 disable、宽泛 ignore 或降低规则级别让 CI 变绿。
- 必须抑制规则时，记录原因、范围、负责人和清理条件。
- 任何新增门禁先观察误报率，再进入阻断模式。

## 12. 持续改进与增量棘轮

历史代码不可能一次达到全部标准。采用“保留基线、禁止恶化、持续收紧”的治理方式。

### 12.1 建立基线

首次接入目标仓库时记录：

- 构建、lint、类型检查和测试现状。
- 测试数量、执行时间和不稳定测试。
- 大文件、大函数、高复杂度热点。
- `any`、忽略规则、deprecated 和 TODO 数量。
- 依赖漏洞、密钥风险和过期依赖。
- 缓存、任务、timer、listener 和内存 Map 的资源边界。
- 线上错误率、延迟、任务积压和资源饱和指标。

基线不是豁免清单。它只用于区分历史债务和新增退化。

### 12.2 增量棘轮规则

- 新代码不得引入新的 P0/P1。
- 修改过的代码不得增加 lint、类型或安全警告。
- 新增分支必须有相应测试。
- 修改热点文件时至少完成一项局部简化，前提是不扩大风险。
- 新增 ignore 或 suppression 必须经过审查。
- 技术债数量不得无理由净增长。
- 指标门槛只能在有证据和批准时放宽。

### 12.3 热点治理顺序

优先级按以下公式判断：

```text
治理优先级 = 变更频率 x 故障影响 x 理解难度 x 并发或安全风险
```

优先处理：

1. 经常修改且经常出错的核心路径。
2. 认证、跨租户、任务、缓存和数据写入。
3. 单函数状态机和多副作用流程。
4. 缺少测试但承担公共契约的模块。
5. 只影响风格、不影响行为的历史问题。

### 12.4 从事故和审查中学习

每个重要事故或重复出现的审查意见都要回答：

- 为什么现有测试、监控和审查没有提前发现？
- 能否转化为静态检查、测试模板或专项 checklist？
- 规则会不会产生大量误报？
- 应加入通用门禁还是特定技术栈规则？
- 是否需要更新示例和反例？

### 12.5 规则生命周期

- 新规则必须有真实案例、风险说明和通过条件。
- 规则先以建议模式试运行，再根据误报率决定是否阻断。
- 每季度检查重复、过时、无法执行或与工具重叠的规则。
- 删除规则必须说明为何风险已经消失或被其他机制覆盖。
- 不允许为了让当前变更通过而临时弱化全局规则。

## 13. 分阶段落地路线

### 阶段 0: 盘点，不改行为

目标：知道当前质量和风险分布。

交付物：

- 仓库结构和关键调用链地图。
- 质量命令清单及实际结果。
- P0/P1 风险列表。
- 高并发、认证、缓存、任务和数据路径清单。
- 前 10 个治理热点。

完成条件：所有结论有文件、命令或运行指标证据。

### 阶段 1: 建立最低门禁

目标：阻止新增退化。

落地项：

- 提供统一的 format、lint、typecheck、test 和 build 命令。
- 建立 PR 模板和标准审查输出。
- CI 阻断新增 lint、类型、构建和测试失败。
- 加入密钥扫描和依赖漏洞检查。
- 高风险变更强制填写专项审查。

完成条件：新问题可被自动识别，P0/P1 有明确阻断机制。

### 阶段 2: 治理核心热点

目标：降低最危险代码的修改成本。

落地项：

- 先为现有行为补 characterization tests。
- 将大型流程拆成状态解析、业务决策、副作用执行和结果呈现。
- 明确状态机、幂等键、锁和资源生命周期。
- 为跨模块契约建立单一 schema 来源。
- 对热路径建立 QPS、扇出和容量模型。

完成条件：热点模块可以独立理解、测试和验证，不依赖一次性全面重写。

### 阶段 3: 扩大自动化

目标：把重复人工意见转成稳定工具。

落地项：

- 增量覆盖率和复杂度报告。
- 架构依赖规则和循环依赖检查。
- API/schema 兼容检查。
- 并发、幂等和故障注入测试。
- 性能基线和资源泄漏测试。

完成条件：自动化检查低误报、执行时间可接受，并有负责人维护。

### 阶段 4: 周期性复盘

目标：让规则随真实风险演进。

建议节奏：

- 每周：整理重复审查意见和不稳定测试。
- 每月：更新热点、质量趋势和技术债净变化。
- 每季度：审查规则有效性、误报、事故覆盖和门禁强度。
- 每次事故后：增加能够提前发现同类问题的最小机制。

## 14. 推荐仓库结构

随着项目演进，优先使用以下结构：

```text
AGENTS.md
docs/
  standards/
    severity.md
    universal-gates.md
    concurrency.md
    security.md
    testing.md
    changelog.md
  playbooks/
    baseline-audit.md
    hotspot-refactor.md
    incident-to-rule.md
  examples/
    good-review.md
    bad-review.md
templates/
  pull-request-template.md
  review-report.md
scripts/
  audit/
  quality-gates/
```

拆分原则：

- `AGENTS.md` 保留必须遵守的核心规则和路由。
- 详细技术栈规则进入 `docs/standards/`。
- 可执行步骤进入 `docs/playbooks/`。
- 可复制格式进入 `templates/`。
- 稳定机械检查进入 `scripts/`。

不要在尚无实际需求时一次性创建所有文件。

## 15. 标准审查流程

### 15.1 审查前

1. 阅读用户目标、issue、设计和目标仓库规则。
2. 检查工作区状态，区分用户改动和当前变更。
3. 阅读 diff 及直接调用链。
4. 标记变更类型和必须执行的风险专项。
5. 确认可以运行的验证命令。

### 15.2 审查中

1. 从输入追踪到输出。
2. 标出所有状态写入和外部副作用。
3. 检查错误、取消、超时、重试和恢复。
4. 检查并发交错和多实例行为。
5. 检查资源容量和清理。
6. 对照测试确认关键分支是否被验证。
7. 区分确定缺陷、风险疑问和非阻断建议。

### 15.3 审查后

1. 运行与变更匹配的检查。
2. 按 P0 到 NIT 排序输出发现。
3. 每条发现提供位置、触发条件、影响和验证建议。
4. 列出开放问题和假设。
5. 简短总结做得好的部分和剩余风险。
6. 没有发现问题时明确说明，并列出未运行测试或未覆盖风险。

## 16. 修改生产代码后的必做复核

任何生产代码修改完成后，必须逐项确认：

- 需求行为已实现，且没有扩展到无关范围。
- 新增或修改路径有可信测试。
- format、lint、typecheck、相关测试和 build 已实际运行。
- 错误、日志和指标没有泄露敏感数据。
- 配置、schema、文档和代码一致。
- 高并发检查已按第 7 节完成。
- 队列、缓存、Map、timer、listener、重试和后台任务有界。
- 多实例、重复请求、失败重试和回滚行为明确。
- 没有通过 ignore、宽泛 `any` 或吞异常绕过问题。
- 未修改或回滚用户无关改动。

如果任一项无法验证，必须在交付总结中明确说明。

## 18. 最终原则

1. 行为正确优先于形式漂亮。
2. 证据优先于直觉。
3. 安全、隔离和数据完整性优先于性能优化。
4. 有界资源优先于流量驱动的无限扩张。
5. 原子 acquire-then-act 优先于 create-then-cleanup。
6. 纯查询优先保持无副作用。
7. 小而清晰的公共接口优先于暴露内部复杂度。
8. 增量改进优先于无计划重写。
9. 自动化处理机械问题，人工审查系统行为和风险。
10. 每次事故和重复审查意见都应让这套体系更强。
